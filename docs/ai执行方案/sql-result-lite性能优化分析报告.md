# `/search/sql-result-lite` 性能优化分析报告

## 一、核心结论

- **当前性能现状**：接口响应时间约 178.5 秒，远未达到 20 秒的目标要求
- **主要瓶颈**：不在数据库执行（约 1 秒），而在多个 LLM 处理阶段
- **优化潜力**：通过结构调整和流程优化，简单查询有机会进入 20 秒内

## 二、问题分析

### 2.1 测试环境与方法

- **测试问题**：帮我统计供水管网中的阀门有多少个
- **测试接口**：POST /api/search/sql-result-lite
- **测试轮次**：
  - 第一轮：带启动期干扰（超时约 180 秒）
  - 第二轮：干净环境复测（耗时 178.5 秒）

### 2.2 性能瓶颈排序

| 处理阶段 | 耗时（秒） | 影响程度 |
|---------|-----------|----------|
| 语义一致性校验 | 56.8 | 最高 |
| 查询增强 | 39.7 | 高 |
| 证据召回 | 37.6 | 高 |
| 意图识别 | 19.9 | 中 |
| SQL 生成 | 19.1 | 中 |
| 表关系分析 | 5.6 | 低 |
| Schema 召回 | 1.1 | 低 |
| 数据库执行 | 1.1 | 低 |

### 2.3 瓶颈原因分析

- **语义一致性校验**：对生成的 SQL 进行完整的 LLM 语义审计，输入信息量大，任务复杂度高
- **查询增强**：需要整合用户问题、业务证据和多轮对话，上下文长度影响处理速度
- **证据召回**：包含 LLM 问题改写和证据拼接，召回内容过多会拖慢后续流程

## 三、优化建议

### 3.1 短期优化（1-2 周）

- **降级语义一致性校验**：对简单聚合查询跳过完整 LLM 审计，改用规则校验
- **压缩证据召回规模**：控制证据拼接长度，避免超长上下文
- **简化查询增强**：对简单问题减少不必要的 query rewrite

### 3.2 中期优化（2-4 周）

- **实现双通道处理**：
  - **快速通道**：针对 COUNT/SUM/AVG 等简单查询，减少处理步骤，目标 10-20 秒
  - **完整通道**：针对复杂查询，保持现有完整链路，优先保证正确性
- **合并处理节点**：将意图识别与查询增强合并，减少 LLM 往返次数

### 3.3 长期优化（1-2 个月）

- **引入模板化生成**：对常见聚合查询使用模板化 SQL 生成
- **智能路由机制**：根据问题复杂度自动选择处理路径
- **性能监控体系**：建立关键节点性能监控，持续优化

## 四、预期效果

| 优化阶段 | 简单查询预期耗时 | 复杂查询预期耗时 |
|---------|----------------|-----------------|
| 当前状态 | 178.5 秒 | 178.5 秒+ |
| 短期优化 | 60-100 秒 | 120-150 秒 |
| 中期优化 | 10-20 秒 | 60-90 秒 |
| 长期优化 | <20 秒 | 40-60 秒 |

## 五、决策建议

1. **认可当前问题**：确认性能瓶颈主要在 LLM 处理阶段，而非数据库执行
2. **支持双通道方案**：为简单查询提供快速路径，复杂查询保证正确性
3. **分阶段实施优化**：先进行短期优化，再推进中期结构调整
4. **设定合理目标**：
   - 短期目标：简单查询进入 60 秒内
   - 中期目标：简单查询进入 20 秒内
   - 长期目标：复杂查询进入 60 秒内

## 六、下一步行动计划

1. **启动短期优化**：优先处理语义一致性校验和证据召回的优化
2. **设计双通道方案**：制定简单查询的快速处理流程
3. **建立性能基准**：设置关键节点的性能监控指标
4. **定期评估效果**：每周评估优化效果，调整策略
